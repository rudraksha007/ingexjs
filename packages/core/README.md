# @ingex/core

The core engine of Ingex. It provides the `Ingestor` class that coordinates crawling and multi-threaded data processing stages.

## Installation

```bash
bun add @ingex/core
```

## `Ingestor` API

The `Ingestor` class is the heart of the pipeline.

### Constructor

```typescript
import { Ingestor } from "@ingex/core";

const ingestor = new Ingestor(storageConnector, config);
```

- `storageConnector`: An implementation of `StorageConnector` (e.g., from `@ingex/mongoose`)
- `config`: Optional configuration overrides:
  - `flushGap`: How many documents to buffer before batch inserting (default: 100)

### Stages & Processing

The `Ingestor` supports multiple stages of execution. 
- The default stage is `crawl`, which triggers the web scrapers.
- Additional stages can be registered and will be executed by Web Workers.

#### `register(stage: string, jobModule: string, workerCount?: number)`

Registers a new processing stage.

- `stage`: The name of the stage (e.g., `'summarize'`, `'filter'`).
- `jobModule`: The path to the file exporting the default worker function. Relative paths will be automatically resolved to absolute paths.
- `workerCount`: Number of concurrent Web Workers to spawn for this stage (default: 1).

```typescript
// Example: Register a stage that cleans HTML content using 4 parallel workers
ingestor.register("clean", "./workers/cleaner.ts", 4);
```

#### `start(stage?: string, applyOn?: string)`

Starts the execution of a stage.

- `stage`: The stage to run (default: `'crawl'`).
- `applyOn`: The name of the previous stage from which data should be fetched. Required for all stages except `'crawl'`.

```typescript
// Start the crawler
await ingestor.start("crawl");

// Start the 'clean' stage, applying it on data generated by the 'crawl' stage
await ingestor.start("clean", "crawl");
```

### Worker Modules Specification

When you register a custom stage, you point to a `jobModule`. This module **must** have a default export that is a function (synchronous or asynchronous). 

The coordinator will send individual data records to this function. The returned value will be saved back into the database under the new stage name.

**Example: `./workers/cleaner.ts`**
```typescript
import type { Data } from "@ingex/core";

export default async function cleanData(item: Data): Promise<Data> {
    // Perform complex operations here
    const cleanedText = item.content.replace(/<[^>]+>/g, '');
    
    // Any extra data added here goes into the 'data' field when using the Mongoose connector
    return {
        ...item,
        content: cleanedText,
        data: {
            wordCount: cleanedText.split(' ').length
        }
    };
}
```

### Hooks & Events

The ingestor emits several lifecycle and job events that you can hook into using `on()` and `off()`:

- `'start'`: Emitted when the crawler or a stage starts.
- `'end'`: Emitted when the stage successfully finishes.
- `'stopped'`: Emitted if the ingestor is paused or stopped manually.
- `'error'`: Emitted on fatal ingestor errors.
- `'jobError'`: Emitted when a specific worker job fails.
- `'jobCompletion'`: Emitted when a single worker job completes successfully.
- `'jobQueueCompletion'`: Emitted when all queued jobs for a batch complete.

```typescript
const hookId = ingestor.on('start', async (data) => {
    console.log(`Pipeline started with ${data.startUrls.length} urls`);
});

ingestor.off('start', hookId);
```
